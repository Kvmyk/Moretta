  - Compare the fetched GitHub tag with the local `VERSION`.
  - If a new version is available, surface this information to the frontend.

- [ ] **Chat History Preservation**
  - Implement logic to store chat sessions (user prompts, AI responses, and metadata) persistently in the database.
  - Add UI functionality to fetch, view, and manage past conversations.

- [ ] **Conversation List & Multi-Model Thread Metadata**
  - Create a persistent conversation index/list view in the database and frontend sidebar/history view.
  - Scope conversations to the authenticated user account, so each user sees only their own threads and history.
  - Store per-conversation metadata such as `conversation_id`, title/summary, creation date, last activity date, selected provider, selected model, and task/file linkage.
  - Persist user ownership fields (`user_id`, username, or OIDC subject) and enforce authorization checks on every conversation/history endpoint.
  - Support conversations that switch between different providers/models over time, preserving which assistant response was generated by which model.
  - Add filtering/sorting in the UI by date, provider, model, and conversation status.
  - Ensure reopening a conversation restores the full message history together with provider/model context for each turn.

- [ ] **SQLite3 -> PostgreSQL Migration Plan**
  - Inventory all current SQLite-backed components: `backend/store.py`, `backend/anonymizer/vault.py`, `backend/config.py`, `data/store.db`, and `data/vault.db`.
  - Decide whether PostgreSQL should fully replace SQLite or be an optional production backend while SQLite remains for lightweight local setups.
  - Introduce a storage abstraction / repository layer so `PersistentStore` and `Vault` are not tied directly to `sqlite3`.
  - Design PostgreSQL schema for `files`, `tasks`, and `pii_sessions`, including indexes for session lookups, TTL cleanup, and history queries.
  - Rework configuration to use explicit DSNs such as `DATABASE_URL` and `VAULT_DATABASE_URL`; update `.env.example`, `README.md`, and `docker-compose.yml`.
  - Add a `postgres` service to Docker Compose and decide whether encrypted blob data remains on disk (`data/blobs`) or moves into PostgreSQL / object storage.
  - Preserve app-level encryption for sensitive fields (`token_map`, blobs), because the current SQLite `PRAGMA key` approach will not carry over to PostgreSQL.
  - Prepare a one-time migration script from existing SQLite databases into PostgreSQL, with backup + rollback steps.
  - Add concurrency tests for parallel uploads/tasks to confirm PostgreSQL actually removes SQLite lock contention under multi-user load.

## Security Hardening & Verification

- [ ] **Full application security review (RCE, prompt injection, exposed ports, misconfiguration)**
  - Run dynamic web/API scans with **OWASP ZAP** against `http://localhost:3000` and the proxied API; use baseline scan in CI and full/API scan in staging. Example: `docker run -t ghcr.io/zaproxy/zaproxy:stable zap-baseline.py -t http://host.docker.internal:3000`
  - Run templated vulnerability checks with **Nuclei** for exposed admin panels, missing headers, default credentials, and known CVEs. Example: `nuclei -u http://localhost:3000`
  - Verify network exposure with **Nmap** and document which ports are allowed in dev vs production; explicitly confirm that `8000` and `11434` are never exposed publicly in production. Example: `nmap -sV -Pn localhost -p 3000,8000,8080,11434`
  - Scan images, filesystem, secrets, and Docker/IaC misconfigurations with **Trivy**. Examples: `trivy fs .`, `trivy config .`, `trivy image privateproxy-backend`
  - Add code-level security scanning with **Semgrep** and **Bandit** for Python/FastAPI and frontend patterns. Examples: `semgrep scan --config auto`, `bandit -r backend`
  - Add secret leakage scanning with **Gitleaks** for both current files and git history. Example: `gitleaks git .`
  - Red-team the LLM flow for **prompt injection, jailbreaks, indirect prompt injection, data exfiltration, and guard bypass attempts** using **Promptfoo** (primary) or **Garak** (secondary). Example: `promptfoo redteam run`
  - Create a repeatable security test matrix: anonymous user, authenticated user, malicious document upload, malicious prompt, poisoned attachment, and internal-only service exposure checks.
  - Produce a hardening checklist from findings: restrict container port bindings, tighten CORS/trusted hosts, validate uploaded files more aggressively, and isolate high-risk services behind the frontend proxy or a reverse proxy.
